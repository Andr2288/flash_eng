# Лабораторна робота №7

## Тема
Проектування архітектури програмної документації та системи забезпечення якості.

## Мета роботи
- сформувати системне уявлення про роль документації в життєвому циклі ПЗ;
- спроектувати набір програмної документації для реального програмного продукту;
- встановити зв'язок між вимогами, архітектурою, інфраструктурою та тестуванням.

## Обраний програмний продукт
`FlashEng` — веб-застосунок для вивчення англійської лексики за допомогою інтерактивних вправ і генеративного ШІ.

---

## 1. Аналіз системи та цільової аудиторії документації

### 1.1 Короткий опис системи
Система `FlashEng` є односторінковим веб-застосунком (SPA), що підтримує персональну практику англійської лексики. Користувач додає слова, фрази або патерни та проходить вправи різних типів. Для генерації навчального контенту застосунок інтегрується з OpenAI API, а для збереження словника і прогресу використовує Supabase. У застосунку реалізовано щонайменше три функціональні сценарії: переклад речень, заповнення пропусків та слухання з введенням відповіді. Логіка відбору наступних слів побудована на статусах (NEW, LEARNING, AGAIN, MISSED) і контрольних точках повторення. Фронтенд розроблено на React, стан керується через Redux Toolkit. Розгортання збірки виконується через GitHub Pages. Система орієнтована на регулярну практику мови та зменшення часу на ручну підготовку вправ.

### 1.2 Формат взаємодії
- основний канал взаємодії: `UI` (веб-інтерфейс);
- зовнішні інтеграції: `OpenAI API`, `Supabase REST/SDK API`.

### 1.3 Цільова аудиторія документації
- **Розробники (Frontend / Fullstack):** архітектура, компоненти, інтерфейси, модель даних, правила оновлення стану.
- **Тестувальники (QA):** функціональні вимоги, сценарії, очікувана поведінка, тестова стратегія, трасування вимог.
- **DevOps/Release Engineer:** інфраструктура розгортання, змінні середовища, CI/CD, надійність, безпека.
  - **Бізнес/Замовник:** призначення системи, межі функціоналу, критерії якості, прогрес реалізації.

## 3. SSD — System Specification Document

## 3.1 Призначення системи
Система призначена для покращення англійської лексики користувачів шляхом:
- збереження словника користувача;
- генерації практичних вправ із прикладами;
- відстеження прогресу та адаптації повторення.

Очікуваний результат для користувача: регулярне повторення релевантних слів із персоналізованим контентом та зворотним зв'язком.

## 3.2 Межі системи

### Входить у межі (In-Scope)
- керування словником (додавання, перегляд, оновлення статусів);
- запуск і проходження 3 типів вправ;
- генерація навчального контенту (речення, підказка, правильна форма);
- генерація аудіо для речень/відповідей;
- базова аналітика прогресу по статусах.

### Не входить у межі (Out-of-Scope)
- повноцінна серверна бізнес-логіка (окремий backend для оркестрації);
- офлайн-режим і синхронізація без інтернету;
- повна рольова модель доступу (RBAC);
- платіжні модулі або монетизація;
- автоматичне перевіряння вимови через ASR.

## 3.3 Функціональні вимоги

### FR-01. Керування словником
Користувач може додати слово/фразу/патерн із необов'язковою темою та релевантними перекладами.

### FR-02. Валідація дублікатів
Система блокує додавання дубліката (порівняння нормалізованого тексту).

### FR-03. Отримання словника
Система завантажує список навчальних елементів із зовнішнього сховища.

### FR-04. Генерація вправи "Переклади речення"
Система генерує українське речення, англійський відповідник і використану форму слова.

### FR-05. Генерація вправи "Заповни пропуск"
Система генерує речення з пропуском, коректну відповідь, переклад і підказку; формує варіанти відповіді.

### FR-06. Генерація вправи "Слухай і впиши"
Система генерує речення та озвучує його; користувач вводить відповідь вручну.

### FR-07. Оцінка відповіді та оновлення статусу
Після відповіді користувача система оновлює статус та checkpoint елемента (REVIEW/AGAIN тощо).

### FR-08. Відбір наступних елементів
Система формує `currentSelection` за пріоритетами (MISSED, NEW, AGAIN, інші).

### FR-09. Показ статистики
Система відображає кількість елементів за статусами для поточного режиму практики.

### FR-10. Навігація по вправам
Система дозволяє перемикати тип вправ, переходити до наступного елемента й керувати сесією.

## 3.4 Обмеження та припущення

### Обмеження
- застосунок працює у браузері та залежить від стабільного інтернет-з'єднання;
- якість контенту залежить від зовнішньої ШІ-моделі;
- швидкодія залежить від затримок OpenAI/Supabase API;
- безпечне використання потребує коректного керування секретами та змінними середовища.

### Припущення
- користувач має сучасний браузер із підтримкою ES2020+;
- зовнішні API (OpenAI, Supabase) доступні та повертають валідні відповіді;
- структура таблиці `vocabulary_words` відповідає поточним полям у клієнтській моделі.

## 3.5 Нефункціональні вимоги

### NFR-01. Продуктивність
- середній час відкриття сторінки практики: до 2 секунд при стабільній мережі;
- час відповіді UI на дії користувача: до 200 мс для локальних операцій стану;
- середній час генерації вправи: до 5 секунд (за нормального навантаження зовнішнього API).

### NFR-02. Доступність
- цільова доступність клієнтського застосунку: не менше 99.0% на місяць;
- graceful degradation при збоях зовнішніх API (повідомлення про помилку без крашу UI).

### NFR-03. Безпека
- секретні ключі не повинні зберігатися у відкритому вигляді в репозиторії;
- доступ до прод-секретів лише через захищені змінні CI/CD;
- валідація введених даних і мінімізація XSS-ризиків у UI-рендері.

### NFR-04. Підтримуваність
- модульна структура коду (components/pages/store/features);
- уніфіковані thunk-потоки для асинхронних операцій;
- окремий документований pipeline збірки та розгортання.

### NFR-05. Масштабованість
- можливість додати нові типи вправ без повного переписування сторінки практики;
- масштабування сховища даних на стороні Supabase.

## 4. SDD — Software Design Document

## 4.1 Архітектурний стиль
Поточний стиль: **клієнтський моноліт (SPA)** з інтеграціями до зовнішніх сервісів.

Підхід:
- React-компоненти для UI;
- Redux Toolkit для стану й бізнес-логіки відбору;
- Async thunks для роботи з API;
- Supabase як кероване сховище даних;
- OpenAI API для генеративних сценаріїв та TTS.

## 4.2 Декомпозиція компонентів

### C1. Routing Layer
- `App`, `Routes`, `Navigate`;
- захист маршрутів через прапорець авторизації (на поточному етапі базовий).

### C2. Presentation Layer
- сторінки: `Home`, `Practice`, `Profile`, `Settings`, `Login`, `SignUp`;
- спільні UI-компоненти: `Navbar`, `Modal`;
- вправи: `TranslateSentenceExercise`, `FillTheGapExercise`, `ListenAndFillTheGapExercise`.

### C3. State Layer
- `vocabularyWordsSlice`:
  - `data` (словник);
  - `exerciseState` (поточний тип вправи, вибірка, індекс, стани завантаження);
  - логіка `makeNextSelection`, `updateExerciseState`.

### C4. Integration Layer
- `supabase.js` — клієнт до Supabase;
- `vocabularyWordsThunks.js`:
  - `addVocabularyWord`, `fetchVocabularyWords`, `updateVocabularyWord`;
  - `generateExerciseVocabularyItem`, `generateSentenceCompletion`, `generateListenAndFill`, `generateSpeech`.

## 4.3 Основні інтерфейси

### UI -> State
- dispatch дій Redux:
  - `updateExerciseState(payload)`;
  - `makeNextSelection()`.

### State -> External API
- Supabase:
  - `insert`, `select`, `update` для `vocabulary_words`.
- OpenAI:
  - `responses.create()` для генерації JSON-контенту вправ;
  - `audio.speech.create()` для генерації аудіо.

## 4.4 Модель даних (логічний рівень)

Сутність `VocabularyWord`:
- `id: number`
- `text: string`
- `topic: string | null`
- `relevant_translations: string | null`
- `status_translate_sentence_exercise: enum`
- `status_fill_the_gap_exercise: enum`
- `status_listen_and_fill_the_gap_exercise: enum`
- `checkpoint_*: number`
- `last_reviewed_*: datetime`
- `created_at: datetime`

Де `enum status` включає: `NEW`, `LEARNING`, `REVIEW`, `AGAIN`, `MISSED`.

## 4.5 Ключові потоки (сценарії)

### Потік A: Додавання слова
1. Користувач відкриває модальне вікно.
2. UI валідовує порожнє поле/дублікати.
3. Викликається `addVocabularyWord`.
4. Дані записуються в Supabase.
5. Стан оновлюється у Redux.

### Потік B: Генерація вправи
1. Користувач обирає тип вправи.
2. `makeNextSelection` формує список поточних елементів.
3. Async thunk звертається до OpenAI.
4. JSON парситься та віддається у компонент.
5. UI відображає завдання.

### Потік C: Оцінка та прогрес
1. Користувач відповідає на завдання.
2. Компонент визначає `isCorrect`.
3. Обчислюється новий checkpoint.
4. Викликається `updateVocabularyWord`.
5. Відбувається перехід до наступної вправи/слова.

## 4.6 Рішення щодо помилок та відмовостійкості
- обгортка `useThunk` для єдиного шаблону `loading/error`;
- показ user-friendly повідомлень при збоях API;
- уникнення краху UI, якщо генерація не повернула валідний JSON.

## 5. ISD — Infrastructure Specification Document

## 5.1 Середовище розгортання
- **Frontend runtime:** браузер користувача;
- **Build system:** Vite (`npm run build`);
- **Hosting:** GitHub Pages;
- **Data layer:** Supabase (cloud PostgreSQL + API);
- **AI services:** OpenAI API.

## 5.2 CI/CD
- GitHub Actions workflow `deploy.yml`:
  - тригер на `push` в `master`;
  - `npm install`;
  - `npm run build`;
  - публікація артефакту в GitHub Pages.

## 5.3 Конфігурація та секрети
- необхідні змінні:
  - `VITE_OPENAI_API_KEY`
  - `VITE_SUPABASE_KEY`
- вимога: у production зберігати в GitHub Secrets, не комітити в репозиторій.

## 5.4 Інфраструктурні компоненти
- **CDN/Static hosting:** сторінки та статичні ресурси;
- **Managed DB/API (Supabase):** читання/оновлення словника;
- **External AI provider (OpenAI):** генерація вправ і TTS;
- **CI runner (GitHub Actions):** автоматизація збірки та публікації.

## 5.5 Принципи масштабування
- горизонтально масштабований frontend через статичний хостинг;
- масштабування API/БД делеговано managed-платформам (Supabase/OpenAI);
- для зменшення вартості й затримок рекомендовано кешування результатів генерації;
- при зростанні навантаження доцільно винести API-ключі на backend-proxy.

## 5.6 Моніторинг і експлуатація (цільовий стан)
- відстеження build/deploy статусів у GitHub Actions;
- логування помилок у клієнті (Sentry або аналог);
- контроль latency зовнішніх API;
- інцидент-процедури для відмов OpenAI/Supabase.

## 6. Документація якості

## 6.1 Test Strategy (коротко)

### Цілі тестування
- перевірити відповідність реалізації функціональним вимогам;
- виявити регресії в логіці прогресу і відбору слів;
- гарантувати стабільність UI при помилках зовнішніх інтеграцій.

### Рівні тестування
- **Unit tests:**
  - редюсери (`makeNextSelection`, `updateExerciseState`);
  - утиліти нормалізації/перевірки відповідей;
  - обчислення checkpoint.
- **Integration tests:**
  - взаємодія компонентів вправ з Redux;
  - thunk-потоки (із моками Supabase/OpenAI).
- **UI/E2E tests:**
  - ключові user journeys (додавання слова, проходження 3 вправ, оновлення прогресу);
  - негативні сценарії (помилки API, порожні стани).

### Типи тестів
- позитивні;
- негативні;
- boundary/corner cases;
- smoke після деплою.

### Критерії входу
- стабільна збірка;
- доступні тестові облікові дані/моки;
- зафіксований обсяг змін.

### Критерії виходу
- 100% виконання критичних тест-кейсів рівня P1;
- відсутність blocker/critical дефектів;
- прийнятний рівень minor дефектів із задокументованими workaround.

### Основні ризики якості
- нестабільність зовнішніх AI-відповідей;
- невалідний JSON від генерації;
- неконсистентність статусів при паралельних оновленнях;
- витік секретів через неправильну конфігурацію.

## 7. Висновки щодо узгодженості SSD-SDD-ISD-Quality

- SSD задає чіткі функціональні рамки навчальної системи та вимірювані NFR.
- SDD деталізує, як ці вимоги реалізуються через SPA-архітектуру, Redux-стан і API-інтеграції.
- ISD описує реальне середовище експлуатації (GitHub Pages + Supabase + OpenAI) і обмеження.
- Test Strategy та Traceability Matrix забезпечують контроль відповідності реалізації вимогам.
- Документація є достатньою основою для подальшого розвитку, тестування й захисту лабораторної роботи.

## 8. Контрольні питання

### 1) Чим SSD відрізняється від SDD?
`SSD` фокусується на тому, **що** система повинна робити (вимоги, межі, обмеження, NFR). `SDD` описує, **як** саме це реалізовано технічно (архітектура, модулі, інтерфейси, потоки даних).

### 2) Яка роль ISD у життєвому циклі ПЗ?
`ISD` визначає, де і як система запускається, оновлюється та масштабується. Він зменшує ризики експлуатації, стандартизує деплой і допомагає відтворювати середовище.

### 3) Навіщо потрібна матриця простежуваності?
Матриця забезпечує зв'язок між вимогами й тестами, дає можливість перевірити покриття, швидко аналізувати прогалини та контролювати регресії після змін.

### 4) Чому документація впливає на якість продукту?
Якісна документація зменшує неоднозначність вимог, прискорює розробку та тестування, полегшує супровід, знижує кількість дефектів і ризиків у релізах.
